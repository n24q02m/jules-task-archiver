## 2024-06-24 - Storage Exhaustion DoS via `CACHE_START_CONFIG`
**Vulnerability:** A malicious or compromised web page could send a massive payload via the `JULES_START_CONFIG` postMessage. This payload would then be proxied to the background script's `CACHE_START_CONFIG` handler and directly written to `chrome.storage.session`. Since `chrome.storage.session` has a strict quota constraint (typically 1MB to 10MB), an attacker could easily exhaust the extension's storage quota, denying service to the application.
**Learning:** `chrome.storage.session` provides shared state storage accessible across the extension. Storing unbounded input directly from potentially untrusted execution contexts (such as isolated worlds interpreting messages from the MAIN world) bypasses capacity limits. Any input leading to persistence APIs must be bounded.
**Prevention:** Always validate and place a strict upper bound (e.g., 50KB or 51,200 chars) on the length of input payloads before processing or persisting them, particularly for data ingested via inter-process communication mechanisms like `postMessage` and `chrome.runtime.sendMessage`.
